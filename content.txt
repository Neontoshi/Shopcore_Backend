## **1. Architecture Overview**

This is a **layered, modular architecture** built with Rust and the Axum web framework, following the classic clean architecture pattern:

```
Handlers (HTTP Layer) → Services (Business Logic) → Repositories (Data Layer) → Database
```

The architecture enforces clear separation of concerns:
- **Handlers**: Accept HTTP requests, validate input, call services
- **Services**: Implement business logic, orchestrate repositories
- **Repositories**: Handle database operations via SQLx
- **Models**: Define data structures that map to database rows
- **DTOs**: Transfer objects for request/response serialization

## **7. Error Handling**

The [app_error.rs](src/errors/app_error.rs) module implements a custom `AppError` enum that maps to HTTP status codes:

```rust
pub enum AppError {
    Database(sqlx::Error)           → 500 Internal Server Error
    Validation(String)              → 400 Bad Request
    NotFound(String)                → 404 Not Found
    Unauthorized(String)            → 401 Unauthorized
    Forbidden(String)               → 403 Forbidden
    BadRequest(String)              → 400 Bad Request
    Conflict(String)                → 409 Conflict
    InternalServerError             → 500 Internal Server Error
    EmailError(String)              → 500 Internal Server Error
}
```

**Features:**
- Automatic conversion from `sqlx::Error` and `anyhow::Error`
- Human-readable error messages returned to clients
- Proper HTTP status code mapping
- Server-side error logging with tracing
- Validation error aggregation

## **8. Middleware Stack**

1. **Auth Middleware** — Validates JWT tokens, extracts user claims, injects `AuthUser` into request extensions
2. **TraceLayer** — HTTP request/response tracing via tower-http
3. **CORS Middleware** — Configured in [cors.rs](src/config/cors.rs)
4. **Rate Limiting** — Via tower-governor (configurable requests/duration)
5. **Compression** — Gzip compression on responses
6. **Request ID Tracking** — Unique ID per request for logging correlation

**Authentication Flow:**
- Client sends JWT in `Authorization: Bearer <token>` header
- Middleware extracts and validates token signature
- Extracts claims (user_id, email, role)
- Injects `AuthUser` struct into request extensions
- Handler accesses authenticated user via extractor

## **9. Request Data Flow**

**Example: Create Order Flow**

```
1. Client POST /api/orders with order data + JWT token
   ↓
2. Router receives request
   ↓
3. Auth Middleware layer
   - Extract JWT from Authorization header
   - Verify signature using secret
   - If valid: inject AuthUser into extensions
   - If invalid: return 401 Unauthorized
   ↓
4. Handler (checkout)
   - Extract State (AppState with db_pool, config)
   - Extract AuthUser from extensions
   - Parse/validate JSON body (OrderRequest)
   - Call OrderService::create_order()
   ↓
5. Service (OrderService)
   - Validate cart items exist
   - Check product availability/stock
   - Calculate totals, tax, shipping
   - Call OrderRepository methods
   - Call AddressRepository for shipping/billing
   ↓
6. Repository layer
   - Execute SQL queries via SQLx
   - Begin transaction
   - Insert order into orders table
   - Insert order items into order_items table
   - Update product stock quantities
   - Commit transaction
   ↓
7. Service returns OrderResponse
   ↓
8. Handler converts to JSON response
   ↓
9. Response sent back to client with 200/201 status
```
